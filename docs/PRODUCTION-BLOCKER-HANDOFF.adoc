== TIER 1 — cheap, not owner-gated, high value (do first)

=== 1.1 Wire `RTSP.inject_rtp` into the SFU broadcast path

*Blocker:* `Burble.Transport.RTSP.inject_rtp/2` is the ONLY function that
feeds media into the RTSP fanout, and it has **zero callers**. So a
stage/broadcast room can accept a viewer, answer DESCRIBE/SETUP/PLAY, and
then send them nothing. Combined with the (now-fixed) dead-on-accept socket
bug, the entire one-to-many streaming path was doubly dead. This is the
single biggest "declared-but-hollow" feature in the shipping server.

*The link already half-exists:* `Burble.Rooms.Room` registers an RTSP
mountpoint for room types `["stage", "broadcast"]` and stores it as
`state.rtsp_mountpoint` (`server/lib/burble/rooms/room.ex:53,127-131`). The
gap is that `Burble.Media.Engine`'s session does NOT know that mountpoint,
so `distribute_to_peers` never calls `inject_rtp`.

*Concrete steps:*
. Thread the mountpoint into the Engine session: pass `rtsp_mountpoint`
  into `Media.Engine.create_room_session/2` (or `add_peer`) and store it on
  the session map (`engine.ex` `handle_cast({:distribute_rtp, ...})` reads
  `state.sessions[room_id]`).
. In `distribute_to_peers/4` (called from the `{:distribute_rtp, ...}`
  handler at `engine.ex:451,452`), after the peer fanout, add:
  `if mp = session[:rtsp_mountpoint], do: RTSP.inject_rtp(mp, packet)`.
. The RTSP side already fans out to subscribers
  (`rtsp.ex:480-490`, sends `{:rtsp_rtp, path, packet}`), so no RTSP change
  is needed beyond confirming subscribers register via `RTSP.subscribe/2`.

*Test:* start a `:broadcast` room, `RTSP.subscribe(mountpoint, self())`,
push a packet through `Engine.distribute_rtp`, `assert_receive {:rtsp_rtp,
^mountpoint, ^packet}`. A hermetic RTSP instance (port 0 + `listener_port/1`,
added in #135) makes this clean.

*Files:* `server/lib/burble/media/engine.ex`,
`server/lib/burble/rooms/room.ex`, `server/lib/burble/transport/rtsp.ex`,
new `server/test/burble/transport/rtsp_broadcast_test.exs`.
*Size:* ~half a day. *Owner-gated:* no. Alternative if descoped: honestly
demote RTSP to experimental in STATE.a2ml + TOPOLOGY.md.

=== 1.2 Resolve the Android BLE detection contradiction (design fix)

*Blocker:* `docs/architecture/ANDROID-CLIENT.adoc` describes two mutually
incompatible detection models and would waste weeks if built as written.
None of it is implemented yet (`client/android/` does not exist; no
`BleSpa.idr`, `nearby-presence.a2ml`, or `ble_spa_verify` in `firewall.zig`)
— so this is cheap to fix NOW, expensive to fix after code lands.

*The contradiction:*

* *Phase 2 (Presence sensor)* has neurophone continuously *read Burble's BLE
  advertisement payload* via a `ScanFilter` on the Burble UUID
  (`ANDROID-CLIENT.adoc:395`, `burble_filter.rs`). This assumes Burble
  *advertises continuously*.
* *Phase 3 (BLE-SPA)* is the silent/secure design: Burble *broadcasts
  nothing* except a one-shot ~2 s HMAC "knock"; "A sees nothing. B sees
  nothing. Neither is discoverable." (`ANDROID-CLIENT.adoc:307-311`).

You cannot continuously detect the presence of something that is silent by
design. **The detection side must be redefined as knock-triggered
rendezvous, not continuous presence sensing.** The neurophone "presence
sensor" as scaffolded will observe only the ephemeral knock, not standing
presence.

*Three Gabeldorsche-specific detection realities to bake in:*
. *Duty cycle vs the "invisible, ~3 mA" claim.* Reliably catching a 2 s
  one-shot knock needs `SCAN_MODE_BALANCED`+ during the expected-knock
  window. The near-zero-battery passive-scan number holds only for the
  *idle* state, not while actively expecting a knock. Document the two
  power regimes separately.
. *"Directed advertisement only to the knocker's MAC" is impractical on
  Gabeldorsche.* With Resolvable Private Addresses and no prior bond you
  don't hold the peer's IRK, so true directed advertising can't target
  them. Realistic design: the responder emits a *second filtered
  non-connectable broadcast* carrying a response token the knocker's app
  matches — not hardware-directed adv.
. *Hardware ScanFilter offload is limited.* The knock rides in
  manufacturer/service data; the controller filter must match on that, and
  Gabeldorsche exposes only a few hardware filter slots — budget them.

*Concrete step:* add a "Detection model" subsection to `ANDROID-CLIENT.adoc`
stating detection is knock-triggered, fold the three realities above into
the BLE-SPA properties table, and re-scope Phase 2 from "continuous presence
sensor" to "knock-observed presence events."
*Files:* `docs/architecture/ANDROID-CLIENT.adoc`.
*Size:* ~1–2 h. *Owner-gated:* the phase-model change (Phase 2 rescope)
should be owner-ratified; the factual corrections are safe to make now.

*RESOLVED 2026-07-09 by ADR-0015 + the BLE wire-format freeze PR.* Detection is
now defined as knock-triggered rendezvous (a normative "Detection model" table
in ANDROID-CLIENT.adoc), the three Gabeldorsche realities are folded into the
BLE-SPA properties, and Phase 2 is rescoped from "continuous presence sensor"
to "knock-observed events (Silent) + contact-resolved presence field (Presence
zones)". Approving that PR's plan ratified the phase-model change.

== TIER 2 — well-scoped, deployment or hook (owner/infra touch)

=== 2.1 Enable authenticated Bolt QUIC

*Blocker:* `:quicer` is commented out in `mix.exs:203`, so
`Burble.Bolt.Quic.available?/0` is always false and every warm/authenticated
bolt path returns `{:error, :quicer_not_available}`. Only raw-UDP bolts run,
and those are trivially spoofable (ADR-0004 acknowledges this). The QUIC
module and a dev cert already exist (`server/priv/cert/bolt.pem`).

*Concrete steps:* re-enable `:quicer` in `mix.exs`; add msquic build to the
`Containerfile` (the reason it was disabled — msquic is hard to build);
add an integration test for knock→verify→CoC with QUIC present.
*Owner-gated:* yes — it's a build/deploy decision (binary size, build time).

=== 2.2 Fix the pre-commit hook owner-grep

*Blocker:* the local `.git/hooks/pre-commit` rejects the repo's OWN header
convention — it greps for `Jonathan D.A. Jewell <j.d.a.jewell@open.ac.uk>`
but every file uses `Jonathan D.A. Jewell (hyperpolymath)
<j.d.a.jewell@open.ac.uk>`. Every commit touching a source file fails; all
three PRs this session used `--no-verify`. This silently trains everyone to
bypass the hook, defeating its purpose.

*Concrete steps:* fix the grep to accept the `(hyperpolymath)` middle
segment; then convert the hook to a *checked-in* hook (a `.githooks/`
directory + `git config core.hooksPath .githooks`) so the fix propagates
instead of living only in each clone's `.git/hooks`.
*Owner-gated:* light — confirm the canonical owner string first.

== TIER 3 — medium builds

=== 3.1 Zig DSP on the live audio path (#52)

The SFU forwards *opaque* Opus; the Zig echo-cancel / comfort-noise / REMB
kernels only receive `record_rtp_timestamp` casts, never actual audio
frames, so they don't run on forwarded media. Also `ZigBackend` is missing
the `decompress_audio_frame/2` behaviour callback. *Decide:* route forwarded
frames through `coprocessor/pipeline.ex`, or document these as receive-side
utilities, not SFU-path features. *Owner-gated:* yes (perf vs. scope).

=== 3.2 End-to-end coverage for the shipped voice client

`client/web/p2p-voice.html` (the actual product) has ZERO direct tests —
every voice/webrtc/signaling test targets separate ReScript modules the page
never imports, and two can't even run because `client/web/src/*.res.mjs`
build artifacts are missing. Add a Playwright two-page harness driving a real
call; fix or regenerate the missing `.res.mjs` artifacts.

=== 3.3 Prove or remove `Burble.LLM.Transport`

The QUIC-8503 failover/endpoint-selector GenServer is real code with zero
live traffic — the REST `/llm/*` path doesn't route through it. Either wire
it into the live path with a test, or delete it and document the single-
endpoint reality.

=== 3.4 Embeddability — drop-in client module

Today embedding = "clone the repo, run a Deno bridge, open an HTML file" —
unusable for a game dev. Extract the P2P voice + AI-channel client into a
single self-contained, versioned JS module exposing a ~5-line API
(`Burble.join(roomCode)`), bridge optional. This is the biggest step toward
"game devs can just use it" and is the natural home for a real demo.

== TIER 4 — owner decision + large / hardware-gated

=== 4.1 AWOL multipath data plane
Control plane exists; the multipath UDP transport (the "never drop a call"
promise) does not — sends return `{:error, :multipath_not_wired}` and the
chaos test is `@tag :skip` documenting the target
(`server/test/burble/network/chaos_test.exs`). *Decide:* implement vs
re-scope. Large build.

=== 4.2 PTP hardware validation
`:ptp_hardware` clock source code + Zig NIF stub exist but are unvalidated —
needs an Intel I210 (or compatible) NIC, not yet arrived. NTP fallback
carries every platform meanwhile; no code change until hardware lands.

=== 4.3 `MediaPipeline.idr` Idris2 proof — RESOLVED 2026-05-20
All 7 ABI proof modules compile under Idris2 0.8.0; the `MediaPipeline.idr`
Idris1 `postulate resampleFrame` was replaced with a pure-Idris2
linear-interpolation definition (see `BURBLE-PROOF-STATUS.md`).
Design-assurance only (no runtime enforcement, ADR-0008). Remaining
proof-layer debt (not a blocker, tracked under epic #53): the
`just build-proofs` recipe still has a path-escaping bug (`cd src\Burble\ABI`
in an sh recipe) and idris2 is not yet wired into CI, so the suite is not
gated.
