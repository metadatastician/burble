== Install-machinery summary (PRs #81 + #85, merged)

The cross-platform service install (now in main):

[cols="2,3", options="header"]
|===
| Platform | Mechanism

| Linux (system, default) | systemd system unit (`/etc/systemd/system/burble.service`) running as `$USER` with `AmbientCapabilities=CAP_NET_BIND_SERVICE` so udp/9 (Bolt WoL poke) binds without root. Requires `sudo` install.
| Linux (user, opt-in) | systemd user unit; `--setcap` flag setcaps the BEAM binary for `udp/9` since user instances can't grant capabilities.
| macOS | launchd LaunchAgent in `~/Library/LaunchAgents/`. Tested via CI on `macos-14` (arm64).
| Windows host (WSL2 NAT case) | True Windows Service via `New-Service -Credential` (runs as installing user so it can see WSL distros). Service exe is a minimal C# `ServiceBase` compiled in-place by the in-box .NET Framework `csc.exe` (no NSSM/srvany dependency). Forwards Bolt udp/7373+9 from host into WSL.
|===

Tests:

* Lint (Linux/macOS/Windows) — render systemd units in both modes + asserts invariants; plist validity; PowerShell AST parse + PSScriptAnalyzer (advisory).
* Round-trip (Linux/macOS/Windows) — actually install→activate→stop→uninstall on the real service manager, with stub `mix`/`deno` binaries so the activate check passes without the full toolchain. Linux + macOS reliably pass; Windows round-trip occasionally exposes platform-specific quirks (documented in PR #85).
