== Topology Semantics (Authoritative Working Map)

Burble runtime currently implements four modes in code:

* `:monarchic`
* `:oligarchic`
* `:distributed`
* `:serverless`

User governance framing adds an explicit `anarchic` concept. For operational alignment:

* Monarchic -> single nexus / single trust domain.
* Oligarchic -> federated trusted nexuses.
* Anarchic -> open plexus behavior; maps primarily to `distributed`, with `serverless` as pure peer mesh extreme.
* Serverless -> no central nexus, E2EE-mandatory mesh.

== Deployment Interpretation: Nexus/Plexus

Operationally useful interpretation for active deployment planning:

* Nexus = a controllable service hub (single or clustered trust anchor) for identity, policy, audit, and room governance.
* Plexus = dynamic interconnection mesh between services/rooms/nodes with explicit capability negotiation and bounded trust.

Using Burble modes:

* Monarchic nexus: local authority, simplest for rollout/testing.
* Oligarchic nexus mesh: trusted federation for game+admin+ops domains.
* Distributed/anarchic plexus: independent authorities with integrity proofs.
* Serverless plexus: peer-first resilience and privacy, minimal server control.
