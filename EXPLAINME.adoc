== 26,350x LZ4 Speedup

[quote, README]
____
SIMD-accelerated audio processing via Zig NIFs
____

Elixir runs on the BEAM VM — brilliant for concurrency, terrible for byte-level
loops. The hot audio path (compression, echo cancel, FFT) runs as Zig
https://www.erlang.org/doc/tutorial/nif.html[NIFs] compiled to SIMD-vectorised
native code. The NIF boundary adds ~200ns overhead.

Same operation, same data. One path is an interpreted byte loop, the other is
compiled SIMD. Run the benchmark yourself: `cd server && mix bench.coprocessor`

**Caveat (ADR-0007):** these are library microbenchmarks of the kernels in
isolation. The live SFU path forwards opaque Opus RTP end-to-end and does not
execute these kernels; they currently serve self-test and archive paths.
End-to-end mic-to-speaker latency is tracked in issue #52.

- Zig implementation: `ffi/zig/src/main.zig`
- NIF interface: `server/lib/burble/coprocessor.ex`
- More on Zig: https://ziglang.org

== Sub-Microsecond PTP

[quote, README]
____
IEEE 1588 PTP (<1us) … Precision timing
____

https://en.wikipedia.org/wiki/Precision_Time_Protocol[IEEE 1588 PTP] is a real
standard used in financial trading, broadcast, and industrial control. We probe
the kernel for hardware PTP clocks, fall back through phc2sys and NTP, and export
jitter as telemetry for the media plane to compensate clock drift.

**Caveat:** Sub-microsecond requires a NIC with PTP hardware clock _and_ a PTP
grandmaster on your network. On a regular cloud VM you get NTP-quality (~1ms).
The code handles this honestly — it reports what it actually has.

- Implementation: `server/lib/burble/timing/ptp.ex` (594 lines)

== Fault Isolation Per Room

[quote, README]
____
Elixir/Erlang (OTP) — auth, sessions, rooms, presence, permissions
____

Each room is a https://hexdocs.pm/elixir/GenServer.html[GenServer] under a
DynamicSupervisor. If a room crashes, the supervisor handles it. Other rooms,
auth, presence, signaling — unaffected. This isn't a framework feature we added;
it's how https://elixir-lang.org[Elixir/OTP] works by design (built for telco
switches needing 99.9999999% uptime).

**Caveat:** OTP isolates _process_ failures, not _infrastructure_ failures. If
VeriSimDB goes down, all rooms lose their backing store.

- Supervision tree: `server/lib/burble/application.ex`
- More on OTP: https://elixir-lang.org/getting-started/mix-otp/supervisor-and-application.html

== Mumble Bridge

Persistent TCP connection to Murmur speaking Mumble's protobuf protocol
(port 64738). Voice (Opus), presence, and text relay bidirectionally. Channel
mapping is configurable.

**What flows now:**

- Server-mute and suppress propagate to Burble — if a Mumble admin mutes a user,
  they are muted in the Burble room too (`UserState` fields 7/9 → PubSub cast)
- Kick and ban events are tracked (`UserRemove` → user removed from phantom list, Burble room notified)
- `PermissionDenied` messages are handled gracefully rather than silently ignored
- Positional audio coordinates (x, y, z) are extracted from incoming Mumble UDP voice
  packets and broadcast on PubSub for Burble's spatial audio. Enable with
  `positional_audio: true` when starting the bridge
- Voice relay uses `UDPTunnel` (TCP) rather than a raw UDP stub, so it actually works

**What still doesn't flow:**

- Full ACL mirroring — Mumble has a 63-permission ACL tree per channel.
  Burble syncs mute/suppress state but doesn't replicate the full ACL model.
- File transfer — not a Mumble protocol feature. Mumble clients share files
  via chat links; there is no `FileTransfer` protobuf message to bridge.
- Positional audio coordinate systems differ per game engine — the x/y/z values
  arrive in whatever space the sender's game uses. A configurable axis remapping
  will be needed for cross-game accuracy.

- Implementation: `server/lib/burble/bridges/mumble.ex`
- Mumble protocol: https://mumble-protocol.readthedocs.io

== SIP Bridge

Subset of https://www.rfc-editor.org/rfc/rfc3261[RFC 3261]. Listens on UDP/TCP
for INVITE/BYE/REGISTER, maps SIP sessions to rooms, transcodes G.711/G.722 ↔
Opus via the Zig coprocessor.

**Caveat:** SIP is a 269-page RFC. We cover common call flows — registration,
basic setup/teardown, DTMF. Complex call transfer scenarios may not work.

- Implementation: `server/lib/burble/bridges/sip.ex` (1,400+ lines)

== Idris2 ABI Proofs & High-Rigor Stack

[quote, README]
____
Idris2 proofs + Zig implementation ... ABI/FFI Design
____

Burble uses a **Dual-Track** architecture. The core Application Binary Interface (ABI) is formally specified in **Idris2** using dependent types. This proves memory layout, protocol transitions, and identity ranks before a single line of runtime code executes.

- **Stack:** Idris2 (Formal Proofs) → Zig (FFI Implementation) → Elixir (Control Plane).
- **Avow:** Consent attestation on room join/leave is proven well-founded using identity ranks.
- **Vext:** Extension sandboxing is proven safe via a capability subsumption lattice.
- **Linearity:** `MediaPipeline.idr` uses linear types to prove that every audio buffer is consumed exactly once, preventing leaks and use-after-free at the type level.

- ABI definitions: `src/Burble/ABI/Types.idr`, `src/Burble/ABI/MediaPipeline.idr`
- Zig FFI: `ffi/zig/src/ffi.zig`

NOTE: An earlier zig REST API layer (`api/v/`) was removed following the zig ban (2026-04-10). The Zig FFI layer (`ffi/zig/`) covers this surface with C-ABI compatibility.

== Zero-Trust SDP & AWOL Routing

[quote, ROADMAP]
____
SDP Foundation ... AWOL Routing
____

Burble replaces traditional firewalls with a **Software-Defined Perimeter (SDP)**. The server is "black" (invisible) until a Single Packet Authorisation (SPA) packet is received. This is handled by a Zig-based `nftables` bridge for near-zero overhead.

**AWOL (Always With One Link)** routing uses the Layline predictive algorithm to handle extreme network conditions (up to 30% packet loss) by predicting RTT velocity and pre-emptively switching paths.

- SDP Implementation: `server/lib/burble/security/sdp.ex`
- AWOL Algorithm: `server/lib/burble/network/awol.ex`

== Burble Bolt

[quote, README]
____
burble bolt <ip> — magic incoming-call packet, like ping but it rings the other end
____

A https://en.wikipedia.org/wiki/Wake-on-LAN[Wake-on-LAN]-inspired magic packet
for voice. Fire it at any IPv4, IPv6, or domain address; if Burble is running
there it flashes a desktop notification and an incoming-call overlay in the
browser — no phone number, no account, just a network address.

The wire format is deliberately WoL-compatible: `"BURB"` magic header, target
MAC repeated 16 times (same layout as a WoL magic packet), then a JSON payload.
This means existing WoL-aware network switches and firmware pass the packet
through without configuration.

Transport priority:

. Raw UDP port 7373 (Burble native, no auth required)
. UDP port 9 simultaneously (IANA discard / WoL port — wake dormant hardware)
. QUIC datagrams (RFC 9221) on port 7373 when quicer is compiled in — TLS 1.3 sender authentication

NAPTR/SRV DNS lookup lets you bolt by domain instead of IP:

[source,dns]
----
example.com. IN NAPTR 10 1 "U" "BURBLE+bolt" "!.*!bolt://198.51.100.1:7373!" .
_burble._bolt._udp.example.com. IN SRV 10 1 7373 burble.example.com.
----

**What happens on the receiving end:**

. `Burble.Bolt.Listener` (UDP GenServer, port 7373) decodes the packet
. `Burble.Bolt.Notify` broadcasts on `Phoenix.PubSub` topic `"bolt:notifications"`
. `BurbleWeb.BoltChannel` forwards to every connected browser tab
. Browser fires the https://developer.mozilla.org/en-US/docs/Web/API/Notifications_API[Web Notifications API] for a desktop pop-up
. Incoming-call overlay appears with Accept / Dismiss buttons
. Accept sends a return bolt to the sender as acknowledgement

**Quick start:**

[source,shell]
----
# Direct IP
bin/burble bolt 192.168.1.100

# IP + MAC (strongest WoL compat — NIC wakes first, then Burble handles it)
bin/burble bolt 192.168.1.100/aa:bb:cc:dd:ee:ff

# IPv6
bin/burble bolt fe80::1

# DNS NAPTR/SRV/A lookup
bin/burble bolt user@example.com

# LAN broadcast (all Burble nodes on the segment)
bin/burble bolt --broadcast

# In development
mix bolt 192.168.1.100
----

- Packet format: `server/lib/burble/bolt/packet.ex`
- Listener: `server/lib/burble/bolt/listener.ex`
- DNS resolver: `server/lib/burble/bolt/naptr.ex`
- Browser channel: `server/lib/burble_web/channels/bolt_channel.ex`

== VeriSimDB & Vext Integrity

[quote, README]
____
VeriSimDB ... handled via Burble.Store
____

VeriSimDB provides native **Provenance** and **Temporal** modalities. Every change to a user account or room config is cryptographically anchored. Because VeriSimDB and the Vext protocol handle internal chain integrity, external blockchain anchoring is redundant and omitted by design.

- Burble Store: `server/lib/burble/store.ex`
- Integrity Logic: `server/lib/burble/verification/vext.ex`
